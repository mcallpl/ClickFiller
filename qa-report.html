The core feature — auto-filling a form — is 100% broken because the API key isn't real

.env / server/analyze.js

Ran the actual capture → crop → "Fill Form" flow with a real test image. The Anthropic API rejected every request with 401 invalid x-api-key — the key in .env was a 22-char placeholder (see-va…), not a real credential.

Effect: every "Fill Form" tap failed, every time. This is the app's entire reason for existing.
Resolved by switching the AI provider to Google Gemini (the owner uses a paid Gemini key, already in the shared vault). What changed:
• server/analyze.js rewritten to use @google/genai (@anthropic-ai/sdk removed). Same prompt, now with schema-enforced JSON (responseSchema) instead of "return only JSON" + regex.
• Key + model loaded from the vault into .env (GEMINI_API_KEY, GEMINI_MODEL); .env stays gitignored.
• Added retry-with-backoff for Gemini's frequent transient 503 "high demand" / 429s.
• Disabled model "thinking" for this extraction task (it was consuming the token budget and truncating the JSON) and raised the output cap.
Model choice: the vault's gemini-2.5-flash-lite returned unusable coordinates (pixels instead of 0–100%, plus empty fields), so ClickFiller is pinned to gemini-2.5-flash via a project-scoped .env override (the shared vault global was left untouched so other projects aren't affected).
Verified live end-to-end: real capture → crop → Fill Form → all 7 fields returned with correct values and rendered on the form (HTTP 200, ~7s). Coordinate positions land in the right area but drift up ~1 row on lower fields — corrected with the built-in drag-to-reposition tool; gemini-2.5-pro would tighten positioning further at higher cost.

Malformed or oversized requests leak a full server stack trace and break the frontend's error handling

server/index.js (no JSON-aware error middleware)

Sent a malformed JSON body directly to /api/analyze. Express has no custom error-handling middleware, so its default handler returned a full HTML page containing the raw Node.js stack trace — including absolute filesystem paths like /Users/chipmcallister/Projects/ClickFiller/node_modules/....

Effect: information disclosure (server file paths, dependency versions) to anyone who sends a bad request. Separately, src/modules/capture.js expects error responses to be JSON and does JSON.parse(text) on them — when it gets this HTML page instead, that parse throws, and the user sees the first 200 raw characters of an HTML error page instead of any real message.
Fixed: added a 4-arg JSON-aware error-handling middleware at the end of server/index.js. Malformed JSON now returns 400 {"error":"Invalid JSON in request body."}, oversized payloads return a clean 413, and any other error returns a generic 500 JSON message — stack traces are logged server-side only, never sent to the client. Re-ran the malformed-JSON curl: response is now application/json with no stack trace. Health check and the missing-image validation path still return correct JSON.

Form fields silently vanish — from both the editor and the exported PDF — with zero indication anything is missing

server/analyze.js (no clamping) · src/canvas-fill.js:156-182 · src/style.css:246

The AI prompt asks for field positions as 0–100% of the image, but nothing clamps that range anywhere in the pipeline — not the server response, not the initial placement (only the drag handler clamps to 0–98%). The result container has overflow: hidden. Reproduced directly: injected a synthetic field at x=104% and another at y=103% alongside a normal one. Measured the live DOM — container right edge was at x=972px, the x=104% field rendered at x=990px (10px past the edge, fully clipped); container bottom was at y=721px, the y=103% field rendered at y=739px (also fully clipped). Screenshot confirms: only the in-bounds field is visible, the other two are completely gone with no error, no warning, nothing.

Effect: any time Claude's vision estimate lands a field near the edge of the form (very plausible on real photos with perspective distortion), that field is detected, computed, and then silently thrown away — before the user ever sees it, and the exported PDF is missing it too since PDF rendering uses the same unclamped coordinates.
Fixed in two places: (1) createFieldOverlay in canvas-fill.js now clamps x/y to 0–98% (matching the drag handler) before placement, so every detected field stays on-canvas and draggable; and (2) server/analyze.js now normalizes the model's response — clamping x/y to 0–100%, fontSize to 0.5–10%, and dropping entries with no value — as defense in depth. Re-verified live: injected fields at x=104% and y=103%; both were clamped to 98% and now render fully inside the container bounds (leftInside: true, topInside: true) instead of being clipped away.

Stored self-XSS via Custom Profile Fields

src/validation.js:199-213, 240-246 · src/modules/profile.js:107-114

Custom field values are only checked for length (≤100 chars) and tag-stripping (removes <...> and javascript:) — neither check touches a bare quote character. Added a custom field with value " onfocus="console.log(...)" autofocus="x, saved the profile, reloaded. Inspected the live DOM: the value attribute is literally value="" onfocus="console.log('CF_XSS_POC_TRIGGERED')" autofocus="x" — the payload broke out of the attribute and injected two new HTML attributes directly into the page, because the row is built with raw template-string innerHTML instead of setting .value as a property.

Effect: this is self-XSS today (only the same user's own saved data triggers it), but it's a real injection primitive sitting in a profile field that also stores SSN and insurance numbers, and it would become a real stored-XSS if profiles are ever synced/shared between users.
Fixed: addCustomFieldRow in src/modules/profile.js no longer builds the row from an innerHTML template string. It now constructs the inputs with document.createElement and assigns the saved strings via the .value property, which treats them as literal text that can never be parsed as markup. Re-verified live: called the real render function with the exact payload " onfocus="..." autofocus="x, then focused the input — nothing executed (xssFired: false), no onfocus/autofocus attributes exist on the element, and the value round-trips back intact. The rendered HTML now has no value= attribute at all.

"Encryption" for sensitive fields doesn't exist — it's a stub that only throws

src/encryption.js (entire file)

Every exported function — encryptField, decryptField, deriveKey, encryptProfile, decryptProfile — is a one-line stub that throws "not yet implemented". Confirmed via grep that nothing in the codebase calls any of these functions, so it doesn't crash anything today — but it also means the "Phase 1" claim that sensitive fields (SSN, insurance provider/policy number) get encrypted before storage is simply not true. All of it sits in plaintext in localStorage, which the profile screen describes only as "Stored only on your device."

Resolved: confirmed no UI text ever claimed encryption and nothing imports the stub, so it was pure dead code with misleading docstrings. Deleted src/encryption.js; production build still succeeds.

Follow-up (not a bug — a product decision for you): if you actually want at-rest encryption, it needs a user-supplied passphrase to derive the key (PBKDF2 → AES-GCM via Web Crypto). Without a passphrase, a client-only app has to store the key right next to the ciphertext, which provides no real protection. That's a feature to design deliberately, so I did not fake it. The profile screen's "Stored only on your device" wording is accurate as-is.

Bonus bugs surfaced once the test suite could run

server/middleware/validation.js · src/validation.js · tests/setup.cjs

With the suite finally executing, 7 tests failed — all pointing at genuine defects, not bad tests:

1. Null/undefined request body crashed the API validator. validateAnalyzeRequest destructured req.body on line 2 before the null-check on line 5, so a null body threw a TypeError instead of returning 400. Fixed by checking the body first, then destructuring. (Covered 3 failing tests.)

2. An exactly-20MB image was wrongly rejected. The size check measured the whole data-URL string, so the data:image/...;base64, prefix pushed an at-limit image over. Now measures the base64 payload only.

3. Accented names were rejected. The name regex [a-zA-Z] rejected valid names like José. Changed to /^[\p{L}'-]+$/u (any Unicode letter + hyphen/apostrophe).

4. Impossible dates were accepted. 2000-02-30 passed validation because new Date() silently rolls it over to Feb 29. Now parses strict YYYY-MM-DD and verifies the parts round-trip.

5. Tests leaked mock state between each other. tests/setup.cjs used jest.clearAllMocks(), which doesn't clear queued mockResolvedValueOnce values — a {fields: []} fetch response from one test leaked into another, causing an order-dependent failure. Switched to jest.resetAllMocks().
